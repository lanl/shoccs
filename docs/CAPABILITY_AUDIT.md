## Two codebases, one repository

SHOCCS is really **two codebases that meet at a single bridge**:

1. **The C++ solver (`src/`).** A Cartesian cut-cell, high-order finite-difference
   PDE solver built on Kokkos. This is the thing that integrates heat / wave /
   (scaffolded) Euler equations in time. It is **migration-complete and
   stabilized**: the range-v3 → Kokkos migration finished, all cleanup plans
   (12, 15, 18, 19, 39, 46) are closed, and 9 of 14 C++ subsystems went dormant
   simultaneously on 2026-03-27. The only recent C++ activity is a narrow
   stencil-installation bridge.

2. **The Python stencil / stability / optimization framework
   (`scripts/stencil_gen/`).** A SymPy symbolic stencil-derivation pipeline plus
   a numeric stability-analysis and parameter-optimization stack (group-velocity
   analysis, an 8-layer Brady-Livescu 2D stability cascade, NSGA-II Pareto
   fronts, multi-fidelity Bayesian optimization). This is where **virtually all
   live development now happens**: of the April 2026 commit surge (~581 commits),
   only ~15 touched C++.

**The asymmetry to internalize as a new developer:** the C++ side is mature but
its *documentation was stale* (the public `readme.md` still advertised range-v3
and a `make` build that no longer exist; the legacy `docs/*.md` design files
predate the registry/Kokkos refactor). The Python side is *less frozen but far
better documented* — `docs/handoff/*` and `scripts/stencil_gen/docs/*` are
current, code-verified, and even self-flag their own stale sections. See the
**[Documentation trust guide](#documentation-trust-guide)** below before
believing any single doc.

The two halves connect through **`cpp_bridge.py`** (Python → C++): the optimizer
renders a Lua config with runtime-parameterized stencil parameters, runs the
compiled `shoccs` binary on the Brady-Livescu 2019 §4.3 2D benchmark, and parses
`logs/system.csv`. With the build fixed (2026-06-04) the `shoccs` binary builds and
runs, so that bridge / L8 real-binary path works again (its smoke test still requires
a built binary).

## Feature-level maturity

### Systems — the most important place to read the fine print

The `systems` subsystem is rated **partial** precisely because "PDE system"
covers five wildly different maturity levels:

- **`heat` — WORKING END-TO-END.** Full eager + Kokkos-graph RHS, MMS
  source/BC handling (Dirichlet & Neumann, grid + object), the most-churned C++
  file in the repo (25 commits), and the *only* system driven through the full
  `simulation_cycle` + integrator stack (`simulation_cycle.t.cpp`,
  `rk4_v2.t.cpp`, `euler_v2.t.cpp` all use `type="heat"`). 61 assertions across
  7 test cases including E2 convergence and graph-matches-eager.
- **`scalar_wave` — IMPLEMENTED, but NOT wired through `simulation_cycle`.**
  Real eager + graph RHS with numeric checks (gradient/dot-product/boundary,
  14 assertions), but it is only ever exercised in its **own unit test** — it is
  never run through the full time-stepping loop. Treat it as "the kernels work,
  the integration path is unverified."
- **`hyperbolic_eigenvalues` — IMPLEMENTED for its narrow purpose, NOT wired
  through `simulation_cycle`.** It is a **diagnostic, not an integrator**:
  `rhs`/`initialize`/`update_boundary` are intentionally empty; `stats()` runs
  the `eigenvalue_visitor` on the gradient operator to get a spectral radius.
  Validated by its own unit test only; never time-stepped.
- **`inviscid_vortex` — EULER SCAFFOLDING ONLY (not a working PDE).** Every
  interface method is an empty stub, `valid()` always returns `false` (so
  `simulation_cycle`'s `while(controller && sys.valid(stats))` loop never
  iterates), `timestep_size` returns a hardcoded `1.0`, there is **no test
  file**, and its analytic `solution::{rho,rhoU,rhoV,rhoE,P}` structs reference
  only each other — dead code. It *is* reachable from `system::from_lua` via
  `type='inviscid vortex'`, which is a trap: the builder will hand you a system
  that does nothing.
- **`empty` — intentional placeholder.** The variant's default alternative; not
  selectable via Lua.

**New-developer takeaway:** SHOCCS today is a *working heat solver* with a
*working-but-unintegrated wave solver*, an *eigenvalue diagnostic*, and *Euler
scaffolding that is not a PDE*. See [systems reference](./reference/systems.md).

### Stencils — production vs. experimental

The `stencil::from_lua` dispatch is mature and is the single object operators
consume. But the *catalog* splits sharply:

- **Production-grade (used by checked-in Lua configs):** `E2` (heat.lua),
  `E2-poly` (scalar_wave.lua, eigenvalues.lua), and `E4u` (the
  `lua-configs/brady_livescu_4_3*.lua` family). These are battle-tested upwind/
  central closures.
- **Installed & unit-tested but with no production config usage:** `E6u`, `E8u`,
  and the cut-cell `E4`.
- **Experimental RBF family:** `tension_E4u_1`, `gaussian_E4u_1`,
  `multiquadric_E4u_1`. These are very recent (plans 42.5/42.6, April 2026),
  research-grade boundary closures. Each unit test asserts the C++ 5×7 boundary
  block matches a Python reference fixture **at exactly one parameter value** —
  no parameter sweep, no cut-cell (`psi`) coverage, no end-to-end use in a
  running system. Their `.t.cpp` companions have churn of only 1–2 commits: the
  newest stencils are the least battle-tested. See
  [stencils reference](./reference/stencils.md).

### Simulation — the real entry point vs. the dead stub

- **The real assembly point is `simulation_cycle::from_lua`.** It owns the five
  top-level components (system, step_controller, integrator, field_io, logs),
  builds the RHS Kokkos graph once, and loops. It is reached in production via
  `app/shoccs.cpp → ccs::simulation_run → simulation_cycle::from_lua().run()`
  and is exercised end-to-end (heat, both RK4 and euler) in
  `simulation_cycle.t.cpp`.
- **`simulation_builder` is a dead stub.** Last touched 2021-05-05; its
  `build()` returns a default-constructed `simulation_cycle`, ignoring the Lua
  table. It is never instantiated, never called. The test only `#include`s the
  header. **Do not use `simulation_builder`** — it looks like an entry point and
  is not one. See [simulation reference](./reference/simulation.md).

### IO — partial cut-cell output, no content tests

`field_io` is fully wired and driven by every system's `write()` and by
`simulation_cycle::run()`. The four IO tests pass today (no Kokkos runtime dep,
so this is a *trustworthy* green). The gaps:

- Tests assert "no exceptions / return values" but **never verify output file
  contents** — the XDMF test comment literally says "one needs to load the
  output in paraview"; the binary data is never read back and asserted.
- The **cut-cell R-point output path** (`field_data::write_geom` and the
  `RX/RY/RZ` sub-grids in XDMF) is driven only by tests passing *empty* R-spans,
  so the cut-cell visualization path is effectively untested.
- `write_every_time` scheduling end-to-end, the 2D dimension-swap branch, and
  the `from_lua` filename/suffix/dir overrides are uncovered.

See [io reference](./reference/io.md).

### Other notable feature-level signals

- **GPU is the conspicuous unfinished thread.** `kokkos_types.hpp` is host-only
  (`DefaultHostExecutionSpace`) and has not changed since 2026-03-10. Plan 14
  (`14-kokkos-gpu`) exists but no subsystem shows GPU churn — planned, stubbed,
  never executed.
- **`divergence` operator does not exist.** Only a commented-out
  `add_unit_test(divergence ...)` line in `src/operators/CMakeLists.txt:20`
  remains; there is no backing source. `gradient`/`laplacian` are real.
- **The Python deterministic core is the reliable ground truth.** The symbolic
  derivation + codegen + PHS regression core runs 351 passed / 5 skipped, fully
  green, in ~13s. The BO/optimizer modules depend on stochastic BoTorch/scipy
  and can flake under `-x`; run them individually, not as a hard gate.

## Documentation trust guide

SHOCCS has both excellent and dangerously-stale docs. Use this map.

### Trust as authoritative

- **`CLAUDE.md`** — the authoritative source for the **C++ side**. Every checked
  claim verified against code (type aliases, Kokkos host-only types, registry
  ownership, expression templates, selection descriptors, C++23-gap polyfills in
  `lazy_views.hpp`, the system variant, all build/test commands, the stencil_gen
  pipeline section).
- **`docs/handoff/*.md`** — the authoritative source for the **Python side**,
  well-maintained (dated 2026-04-15 → 04-30) and code-verified:
  - `MASTER.md` — entry point; L1–L8 stability cascade, plan 40–47 status, CLI.
  - `framework_architecture.md` — accurate C++ stencil table + `from_lua`
    dispatch (matches `src/stencils/` exactly).
  - `operating_conventions.md` — C++ build/run commands all match reality.
  - `scientific_findings.md` — knowledge results (tension-spline fails BL42
    universally; BL §4.2 eigenvalues; the h-scaling bug fix), internally
    consistent.
  - `completed_plans.md` / `known_limitations.md` / `next_steps.md` — current
    **with self-documented staleness**: each has a top-of-file "Snapshot status"
    disclaimer telling you which of its own sections to distrust. Responsible doc
    hygiene — believe the disclaimers.
- **`scripts/stencil_gen/docs/*`** and **`docs/brady2d_cpp_bridge_reference.md`**
  — current, consistent with `cpp_bridge.py` and the real binary path.
- **`docs/reference/*.md`** (this audit's companion subsystem deep-dives) — the
  per-subsystem references linked in the matrix above.

### Do NOT trust for current API/implementation (STALE)

- **`readme.md`** — the public front door and the **most misleading file**. Still
  advertises range-v3 and a `cd build && make` flow; both are gone (Kokkos +
  CMake/Ninja/spack now). Omits Kokkos/Boost/lapackpp/benchmark from its
  dependency list. Highest-value single doc fix.
- **`docs/field.md`** — describes `scalar_field`/`vector_field` types that no
  longer reflect the registry-based `field_registry`/`field_ref` API.
- **`docs/lazyness.md`** — literally says "We'll stick with `range-v3`"; directly
  contradicted by reality (range-v3 removed, utilities reimplemented in
  `lazy_views.hpp`).
- **`docs/expression_templating.md`** — an aspirational DSL wishlist
  (`"du/dt = u.div(u)"_solver`) that was never implemented; the shipped form is
  `dst = a + alpha*b` in `src/fields/expr.hpp`.

### Treat as pre-Kokkos design-rationale archives (PARTIALLY-STALE)

Concepts hold; interfaces, types, and struct members are pre-Kokkos and partly
wrong. Useful for *why*, not for *current API*:

- **`docs/design.md`** — high-level F-vs-R-point philosophy still valid;
  implementation details dated.
- **`docs/discrete_operators.md`** — math ($O^x + B^x$ split, CSR boundary
  handling) current; the proposed `operator_builder` C++ interfaces do not match
  the shipped registry API.
- **`docs/matrices.md`** — taxonomy (dense corner / circulant interior / block /
  CSR) matches; the range-v3-style lazy `operator*`/`operator+` signatures are
  stale (current code uses explicit-loop MatVec).
- **`docs/stencils.md`** — conceptual split current; predates the generated
  RBF/PHS stencils and `from_lua` runtime dispatch.
- **`docs/geometry.md`** — most accurate of the legacy set; ray-casting/`hit_info`
  description holds, but the documented `mesh_object_info` is missing the
  `normal` field the real struct has.
- **The root migration spec
  (`SHOCCS_ARCHITECTURE_AND_KOKKOS_MIGRATION_SPEC.md`)** — a *pre-migration*
  planning document whose **premise is inverted** (it describes the range-v3 →
  Kokkos migration as future work; it is done). Its subsystem decomposition and
  PDE/method overview remain useful, but its range-v3 code examples,
  migration roadmap, the non-existent `cc_elliptic` system, the wrong
  `mesh_object_info` members, and the incomplete stencil list are all obsolete.
